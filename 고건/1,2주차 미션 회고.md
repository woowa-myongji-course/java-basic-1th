# 객체지향 미션 회고

## 1. 계산기 미션 회고

### 단위 테스트를 통한 설계 경험과 리뷰 피드백
계산기 미션에서는 단위 테스트를 작성하면서 테스트가 설계에 미치는 영향을 실제로 체감할 수 있었습니다. 테스트 케이스를 꼼꼼하게 작성하는 과정에서 입력값의 예외 범위, 예외 처리 방식, 입력값 가정 등에 대해 고민하게 되었습니다.

미션 피드백을 통해  
- 예외 처리 범위를 논리적으로 정의하는 법  
- IllegalArgumentException과 같은 구체적인 예외 타입을 선택하는 이유   

같은 부분을 배우게 되었습니다.

또한 개발자 관점과 사용자 관점을 구분해서 설명해 주셨는데, 시스템 사용자 관점에서는 입력값의 합리적인 범위가 필요하고, 개발자 관점에서는 의도치 않은 파라미터 입력에 대한 예외 처리도 고려해야 한다는 점을 이해할 수 있었습니다.

이 경험을 통해 단위 테스트는 단순한 검증을 넘어서  
설계의 명확성, 책임 분리, 개발자와 사용자 관점의 균형을 만드는 중요한 도구라는 점을 확실히 느꼈습니다.

---

## 2. 레이싱카 미션 회고

### 테스트 가능성을 고려한 구조 설계
레이싱카 미션에서는 테스트가 어려운 구조가 어떤 문제를 만드는지 명확하게 느낄 수 있었습니다.

랜덤 요소가 포함된 로직은  
그 자체로 테스트를 불안정하게 만들었고,  
이를 해결하기 위해 외부에서 제어 가능한 구조가 필요하다는 점을 깨달았습니다.

이 과정에서 테스트 가능성을 높이기 위해  
의존성을 분리하고 주입하는 방식의 중요성을 체감할 수 있었습니다.

---

### 테스트 코드에서의 상속 사용과 설계에 대한 고민
레이싱카 미션에서 상속을 사용한 경우는  
구현 코드가 아니라 테스트 코드에서 FixedRandom을 사용했을 때였습니다.

구현 단계에서는 상속을 설계 선택지로 깊게 고려하지 않았지만,  
테스트를 작성하면서 상속을 사용해 랜덤 값을 고정하는 방식에는  
명확한 한계가 있다는 점을 느끼게 되었습니다.

테스트를 위한 하위 클래스가 생기면서  
도메인 설계와 테스트 코드의 경계가 흐려질 수 있고,  
상위 클래스의 변경이 테스트 코드에 영향을 줄 수 있다는 점이 드러났습니다.

이후 조합을 사용하여  
움직일 수 있는지 판단하는 정책을 분리하고,  
테스트에서는 해당 정책만 교체할 수 있도록 구조를 변경했습니다.

이를 통해  
- 상속은 테스트 편의를 위해 사용될 경우 구조를 왜곡할 수 있으며  
- 동작이나 정책을 교체하려는 목적이라면

조합이 더 적합한 선택이 될 수 있다는 점을 체감했습니다.

---

### 의존 방향에 대한 인식
또한 레이싱카 미션을 통해 의존 방향에 대해서도 고민하게 되었습니다.

GameManager와 같은 상위 레벨 객체가  
구체적인 구현이나 세부 로직에 직접 의존하게 되면  
작은 변경에도 영향 범위가 커질 수 있다는 점을 느꼈습니다.

이에 따라 상위 정책은 추상화에 의존하고,  
구체 구현은 외부에서 주입받는 방향으로 구조를 조정했습니다.

이 경험을 통해  
의존 방향은 항상 변경 가능성이 적은 쪽을 향해야 하며,  
테스트를 위한 구현 역시 도메인 바깥에서 교체 가능해야 한다는  
설계 기준을 인식하게 되었습니다.

## 3. 클라이언트를 바라보는 두 가지 관점

미션을 진행하면서 동일한 코드를 두 가지 관점으로 바라보는 경험이 중요함을 느꼈습니다.

### 개발자의 관점
- 역할과 책임이 분리되어 있는가
- 테스트가 작성하기 쉬운 구조인가
- 변경이 발생했을 때 수정 범위가 제한적인가

를 중심으로 고민하게 되었습니다.

### 사용자의 관점
- 입력 형식이 직관적인가
- 잘못된 입력에 대해 예외 메시지가 이해 가능한가
- 사용자가 실수했을 때 시스템이 어떻게 반응하는가

에 대해 생각해볼 수 있었습니다.

특히 입력 형식과 예외 상황을 테스트로 표현하면서  
개발자에게는 단순한 문자열 입력이  
사용자에게는 하나의 인터페이스가 될 수 있다는 점을 인식하게 되었습니다.

## 4. 앞으로의 학습 방향

앞으로 미션을 진행하면서 다음과 같은 기준을 가져보려고 합니다.

- 구현 전에 단위 테스트 작성 범위 먼저 정리하기

- 테스트가 어려운 부분이 있으면 설계 개선을 고려하기

- 객체 설계 시, 코드 작성자의 관점뿐만 아니라 코드를 사용하는 사용자의 관점도 함께 고려하기  

- 랜덤, 시간, 외부 환경과 같이 테스트를 어렵게 만드는 요소를 외부에서 제어 가능하도록 분리하기  

- 테스트를 위한 구현이 도메인 설계에 영향을 주지 않도록 하기

- 상위 레벨 객체가 구체적인 구현이 아닌 추상화에 의존하도록 의존 방향을 항상 점검하기  

- 기능 추가나 정책 변경 시 기존 테스트를 깨뜨리지 않고 확장 가능한 구조를 목표로 설계하기
